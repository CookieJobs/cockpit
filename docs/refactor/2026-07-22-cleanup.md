## #1 抽 Popover 基础组件 (StatusMenu / PriorityMenu 共用)

### 动机
- `web/components/MainBoard.tsx` 的 `StatusMenu` (line 1393) 和 `PriorityMenu` (line 1291) 两个组件**几乎一字不差**地写了 ~30 行 useState + useRef + click-outside mousedown 监听 + Esc 监听 + focus 回到 trigger 的样板代码

- 两个组件 Esc 行为**不一致**: StatusMenu 关闭后 focus 回去 trigger, PriorityMenu 不 focus

### 改动
- **新文件** `web/components/Popover.tsx` (79 行) — 暴露 `usePopover()` hook, 状态: `{ open, setOpen, toggle, close, containerRef, triggerRef }`
- 行为 1:1 等价于原内联实现, 统一为 "Esc 关闭 + focus 回到 trigger" (StatusMenu 原行为, 顺带修了 PriorityMenu 的小不一致)
- `MainBoard.tsx` 里 `StatusMenu` 和 `PriorityMenu` 各减 ~30 行, 改用 `const pop = usePopover()` 替换所有 `useState` / `useRef` / `useEffect` 样板

### 验证
- `npx tsc --noEmit` clean
- 3 个 popover 调用点 (`PriorityMenu` 渲染 / `StatusMenu` 渲染 / `StatusMenu` 各项 button) 行为不变
- **2 个新 invariant** (`test_complete_path_invariants.py::test_popover_hook_exists_in_separate_file` + `test_statusmenu_and_prioritymenu_use_shared_popover_hook`) 锁住"必须共用 usePopover", 防回退到内联

### 风险

- 一个**小幅 UX 升级** (PriorityMenu Esc 也 focus 回去), 跟 StatusMenu 保持一致, 不是回归

### 关键不变量

## #2 ORM 抽 mixin (5 个 mixin, 4 个 ORM 简化)

### 动机
- 6 个 ORM 类 (`ProjectORM` / `TaskORM` / `AchievementORM` / `SettingsORM` / `ChatSessionORM` / `ChatMessageORM`) 全部重写 `id: Mapped[str] = mapped_column(String(64), primary_key=True)` boilerplate
- 时间戳字段 (`created_at` / `last_active_at` / `updated_at`) 4 处 ORM 重复, 类型 (date vs datetime) 容易混
- 未来加新表要重抄一遍, 字段名/类型不一致 (lesson #10 教训: `description` 漏传就是 ORM model 字段名不显眼)

### 改动
**新 5 个 mixin** (`app/core/storage.py:48-110`):
| Mixin | 字段 | 用途 |
|---|---|---|
| `StringPKMixin` | `id: str PK` (String(64)) | 4 个 ORM 用 |
| `CreatedDateMixin` | `created_at: date` (default=date.today) | Project, Task |
| `CreatedDateTimeMixin` | `created_at: datetime` (default=datetime.now) | ChatSession, ChatMessage |
| `LastActiveDateTimeMixin` | `last_active_at: datetime` (default + onupdate) | ChatSession |
| `UpdatedDateTimeMixin` | `updated_at: datetime` (server_default + onupdate) | Settings |

**4 个 ORM 简化** (SettingsORM / AchievementORM 因为字段名 (`key` / `date`) 语义不同保留手写, 反而**保持语义清晰**):
```python
# 改前
class ProjectORM(Base):
    id: Mapped[str] = mapped_column(String(64), primary_key=True)
    name: Mapped[str] = mapped_column(String(200), nullable=False)
    description: Mapped[str] = mapped_column(Text, default="", nullable=False)
    created_at: Mapped[date] = mapped_column(Date, nullable=False, default=date.today)
    archived: Mapped[bool] = mapped_column(Boolean, default=False, nullable=False)

# 改后
class ProjectORM(StringPKMixin, CreatedDateMixin, Base):
    name: Mapped[str] = mapped_column(String(200), nullable=False)
    description: Mapped[str] = mapped_column(Text, default="", nullable=False)
    archived: Mapped[bool] = mapped_column(Boolean, default=False, nullable=False)
```

### 设计决定
- **不**把所有 ORM 全部走 mixin — `AchievementORM.date` (业务日期) 和 `SettingsORM.key` (key-value) 语义不同, 抽反而损失清晰度, 写 docstring 解释
- Mixin 顺序是 `Mixins..., Base` (不是 `Base, Mixins...`) — SQLAlchemy 2.x 要求 mixin 在 Base 前, 让 Base 的 metadata 收集 mixin 字段
- `LastActiveDateTimeMixin` 显式注释为什么用 Python 端 `datetime.now` 而非 `server_default=func.now()` (lesson #8 教训: async SQLAlchemy MissingGreenlet)

### 验证
- 141 → 141 全过 (storage 行为零变化, 表结构零变化)
- 0 个新 invariant 需要 (存储层测试本身就锁住了)

### 风险
- **极低** — 纯重构, ORM 行为和表结构完全不变
- MRO 顺序如果写错会丢字段 — 通过 storage 测试 + Lesson #3 端点级测试两道防线

## #7 setup.sh 末尾加 smoke test (uvicorn + curl + kill)

### 动机
- Lesson #8 明确说"setup 跑完 ≠ 服务能起": greenlet 漏装 / FastAPI NameError 等隐式坑, 装依赖通过但服务起不来
- 现状: 跑 `make setup` 完只 echo ✅, 用户得跑 `make all` 才看到端口 0 监听
- 修法: setup 末尾加 `uvicorn + curl /api/health + kill`, 跑过才算 ✅

### 改动
- `scripts/setup.sh` 加 `[6/6]` 步骤:
  ```bash
  .venv/bin/uvicorn app.main:app --host 127.0.0.1 --port $SMOKE_PORT --log-level warning &
  SMOKE_PID=$!
  for i in 1..10; do
    sleep 1
    curl -sf http://127.0.0.1:$SMOKE_PORT/api/health >/dev/null && SMOKE_HEALTHY=1 && break
  done
  kill $SMOKE_PID 2>/dev/null
  ```
- 失败时输出日志 + 警告 (不阻塞整个 setup exit, 用户已经装好环境了)
- 端口用 `COCKPIT_PORT` 环境变量, 默认 7842

### 设计决定
- **失败不让整个 setup exit non-zero** — 警告即可, 用户可以手动 `make dev` 重试
- **日志输出到 /tmp/cockpit-smoke.log** — 失败时 tail 20 行让用户看到根因, 不丢上下文
- **端口从 `COCKPIT_PORT` 读** — CI / 多实例场景可覆盖, 不写死
- **最多 10 秒** — 正常启动 1-3 秒, 10 秒足够, 也不让 setup 卡太久

### 验证
- 实际跑 `bash scripts/setup.sh`: `✅ 后端能正常启动 (http://127.0.0.1:7842/api/health 响应 ok)`
- 148 测试全过 (setup 改动不进入 pytest 范围)

### 风险
- **极低** — 后台起 uvicorn + kill, 失败只警告
- 一个**未来改进**: smoke test 也可以起前端 next dev 检查 (但 next dev 启动慢, 不在 setup 范围)

## 📋 决策记录 (为什么这么改 / 不改)

| 决策 | 理由 |
|---|---|
| `SYSTEM_PROMPT` 改名字保持 | `run_chat` / `run_chat_stream` 调用点零改动, diff 最小 |
| `AchievementORM` / `SettingsORM` **不**用 mixin | 字段名 (`date` / `key`) 语义不同, 抽 mixin 反而损失清晰度 |
| Popover 抽 hook 而非组件 | StatusMenu / PriorityMenu 触发 button 视觉差异大 (圆点 vs 横条), render prop 反而更绕, hook 够用 |
| main.py 用模块顶层 `register_static_routes(app)` 而非 lifespan 里调 | SPA catch-all 必须在所有 API router 之后注册 (FastAPI 路由按声明顺序匹配), 跟 lifespan 无关 |
| smoke test 失败不阻塞整个 setup | 警告即可, 用户已装好环境, 阻塞会让他跑 make 都跑不了 |
| test_static_serve_invariants 扫描位置改 main.py → main_static.py | 测试应跟代码走, 抽了模块测试也要更新 |
| 不引 E2E (playwright) | 跟"不装乱七八糟"原则冲突, 148 个不变量 + 端点级测试覆盖 80% 静默 bug |
